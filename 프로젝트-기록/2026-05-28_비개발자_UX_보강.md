# 2026-05-28 — 어플 비개발자 사용성(UX) 보강 (Phase 1~4)

## 배경
기능(자동 추출·케이스 분류·인쇄 PDF)은 거의 완성됐지만, **실제 사용자(비개발자·바쁜 영업담당)** 관점에서
"내가 확인할 곳을 못 짚어줌 / 인쇄 전 안전장치 없음 / 미리보기가 작아 검수 불가 / 작업이 쉽게 날아감 /
다운로드 마무리 안내 없음 / 개발자 문법(`{강조}`) 노출"의 6가지 마찰이 남아 있었다.
사용자가 6가지 모두 보강 요청 → 효과 큰 순서로 4단계(Phase) 구현.

## ✅ 오늘 한 것

### Phase 1 — 확인 가이드 + 인쇄 전 안전장치
- **스텝칩 "확인필요" 배지**: `stepIssueCount(id)` 가 `state._extraction.fields` 에서 medium/low/missing 인
  편집가능 항목을 단계별로 셈 (basic→aptName/period/stores/notices, table→benefitTable). 주황 숫자 배지.
- **신뢰도 테두리 확장**: `applyConfidenceStyles()` 가 기존 aptName/period/stores 외 **유의사항·표 제목**도 색칠.
- **완료 단계 요약 체크리스트**(`renderDoneSummary`): 아파트명/기간/매장/표 행수/결제/디자인 한눈에. 빈값은 빨강.
- **인쇄 전 점검**: `measureOverflow()`(배너 클론을 height:auto 로 펼쳐 1800px 초과분 측정) + `emptyRequired()`.
  - 미리보기 툴바에 "내용 넘침" 빨강 칩(rerender 마다 갱신), 완료 요약 상단 경고, 다운로드 직전 빈값 confirm.

### Phase 2 — 미리보기 검수 도구
- 툴바에 줌 컨트롤 [기본 0.8 / 100% 실제크기 / 200%] + [전체화면]. `setZoom(scale)` 이 #bannerRoot transform + .preview-frame 폭 동시 조절.
- **전체화면 오버레이**: 화면 높이·폭에 맞춰 배율 자동 계산, Esc·배경클릭·닫기버튼으로 해제.

### Phase 3 — 작업 보호 (자동 저장)
- `localStorage['iloom-xbanner-draft']` 에 {state, stepId, savedAt} 저장(rerender 시, `allowDraftSave` 로 boot/복원 중엔 끔).
- 시작화면에 **"이어서 편집"** 버튼(드래프트 있을 때만, 마지막 저장시각 표시) → 새로고침/실수 종료에도 작업 보존.
- 파괴적 동작 확인: "기본값 복원"·"새 배너"는 편집이 있을 때만 confirm + 드래프트 삭제. ("← 처음"은 데이터 안 지움 → 확인 없음).

### Phase 4 — 다운로드 마무리 + 개발자 문법 숨김
- 메인 상태줄(`#mainStatus`): 생성 중 "인쇄용 PDF를 만들고 있어요…(최대 10초)" + PNG/PDF 버튼 동시 비활성화 →
  완료 시 "다운로드 폴더의 **{파일명}** 을 인쇄소에 보내세요." (실패도 같은 줄에 표시, alert 폐지).
- 프로모션 헤드라인의 `{강조}=코랄` 문구 제거 → **"선택한 글자 강조" 버튼**(드래그 선택을 `{ }`로 감쌈). 렌더는 기존 `{}`→`<strong>` 유지.

## 🎨 추가 — 편집 UI 라이트 재단장 (모어 쇼룸 톤)
사용자가 모어 쇼룸 웹(`series-모어/쇼룸_웹/index-v3.html`)을 "좋아하는 UI"로 지목 → 다크모드였던 편집화면을
**쇼룸 톤(라이트)** 으로 전면 재단장. 결과물 배너(크림+코랄)와도 톤이 일치.
- `editor.css` 전면 재작성 — `:root` 토큰 도입: 배경 크림/화이트(`--bg #FAFAF8`/`--surface`), 글자 검정·회색(`--ink #2A2A2E`/`--ink-soft`/`--ink-mute`), 선 `--line #E5E3DD`, **강조 코랄 1색 `--hot #C84A3F`**(주요 액션·active·주의 전용). 위계는 색이 아니라 굵기·크기로(쇼룸 디자인_가이드 준수).
- 신뢰도 테두리: "확인 필요"에만 코랄(medium=옅은 코랄, low=진한 코랄), high는 강조 없음 → 색 노이즈 감소.
- 상태 토스트만 기능색 예외(busy=중립, ok=옅은 올리브그린, err=코랄).
- **툴바 정돈**: `.btn-ghost` 의 `width:100%` 제거 → "← 처음" 버튼이 풀폭으로 늘어나던 버그 해결. case 배지는 중립 회색으로 차분하게.
- 사용자 추가분(같은 세션, 별도): 이미지 해상도 점검(저DPI 경고 → 완료 경고에 합류) + 완료 단계 "인쇄 발주 정보" 섹션 — 새 클래스(`.order-spec/.order-title/.order-auto/.order-row/.order-check`)도 라이트 톤으로 스타일.
- 검증: verify-ux 24체크 재통과(에러 0) + 라이트 UI 스크린샷 육안 확인(기본/프로모션/완료 단계).
- 메모리 등록: [[feedback_xbanner_editor_light_theme]] (이 앱은 다크 표준 예외 — 라이트 유지).

### 단계 표시기 재설계 ("단계 느낌이 약하다" → 이후 "깔끔한 progressbar 선호")
기존 알약 칩이 두 줄로 흩어져 순서·진행감이 약했음. 1차로 연결선+번호 동그라미 스텝퍼를 만들었으나,
사용자가 참고 이미지(ui_gradient "Types of Progress Bar" #3 — 라벨+%+얇은 둥근 바)를 보여주며 **더 깔끔한 진행 바**를 선호 → 그 형태로 확정.
- **헤드**: 좌측 "진행 단계" · 우측 "n / 전체"(현재 숫자 코랄).
- **얇은 둥근 진행 바**(`.wiz-track`/`.wiz-fill`) — 폭 = (현재+1)/전체 %, 코랄 채움, width 트랜지션.
- **바 아래 단계 이름**(`.wiz-l`, 텍스트만) — 현재=코랄/지난=진하게/남은=흐리게, 클릭 시 해당 단계로 점프. "확인 필요"는 이름 옆 작은 코랄 점(`.wiz-dot`).
- `renderStepNav()` 를 진행바 구조로 교체. 클릭 핸들러 `.wiz-l`, verify-ux/스크린샷 선택자(`.wiz-l`/`.wiz-dot`) 동기화. 4단계(case-a)/5단계(case-b) 검증.

### max-benefit 빈공간 자동 채움 ("케이스 고정 말고 배너별로" 피드백)
사용자가 대구범어(case-c) 배너의 "최대 혜택"이 다른 배너보다 작다고 지적 → 원인은 case-c 고정 폰트 오버라이드(24/30/22, 표준은 35/40/35). 사용자: **"케이스가 아니라 그 배너의 빈자리에 맞춰 채워야 하고, 키워도 디에이치와 커플링되면 안 된다."**
- 신규 `app/public/fit-banner.js` `fitMaxBenefit(banner)` — 렌더 직후 배너 클론을 height:auto로 펼쳐 콘텐츠 높이 측정, max-benefit을 v8 표준(scale 1.0)~최소(0.62) 중 **그 배너에 맞는 최대 배율**로 설정(1800 안에 들도록). 배너마다 독립.
- `banner.css`: max-benefit 폰트 `calc(var(--fs-mb-*) * var(--mb-scale,1))` 로 변경 + **case-c 고정 오버라이드 제거**.
- 배선: app.js rerender/openFullscreen/boot(폰트 로딩 후) + preview-app.js(내보내기, fonts.ready 후). 양쪽 동일 동작.
- 검증(`scripts/verify-fit.mjs`): 대구범어 scale 1.0(main 40px·콘텐츠 1687px·안넘침) / 디에이치 scale 0.62(main 24.8px). **둘 독립** ✓. verify-ux 24체크 회귀 없음.
- 🔴 발견: 디에이치는 max-benefit 최소로도 152px 넘침(제품카드3+표+결제+유의사항6이 1800 초과) — 기존부터 있던 밀도 문제. "내용 넘침" 칩이 경고. 완전 해결은 [[feedback_xbanner_auto_fill_blank_space]]의 "다른 영역(결제/유의사항) 압축" 구현 필요(미착수).

### 결제 안내 표시 방식 선택 ("로고만/작은 카드 등 다른 모양새도 있잖아" 피드백)
3가지 결제 모드(풀 카드 card_1 / 작은 카드 card_2 / 로고만 compact)가 렌더엔 이미 있었으나 **케이스마다 하나로 고정**(A=풀카드, B·C·E=로고만, D=작은카드)이라 사용자가 못 골랐음 → max-benefit 때와 같은 원리로 **사용자 선택**으로 전환.
- 신규 `render/components/payment.js` `renderPayment(data, defaultMode)` — `data._payMode`(사용자 선택) 우선, 없으면 케이스 기본값. 5개 케이스(a~e) index.js 가 이 단일 진입점 호출하도록 교체(기존 PaymentCards/PaymentCompact 직접 호출 제거).
- 편집 UI: 결제 단계에 "표시 방식" 3버튼(풀 카드/작은 카드/로고만) — `renderPayModes()`, 클릭 시 `state._payMode` 설정 후 즉시 미리보기 반영. `index.html`/`editor.css`(`.pay-modes`/`.pay-mode`) 추가.
- card_2(`.point-card.compact`)·compact(`.payment-brands-compact`) CSS 는 케이스 무관 전역 → 어느 케이스에서도 정상.
- 검증(`scripts/verify-paymode.mjs`): 풀카드↔로고만↔작은카드 전환 시 미리보기 DOM 정확히 변경 8/8 + verify-ux 24체크 회귀 없음.
- 참고: [[feedback_xbanner_auto_fill_blank_space]] SPEC 의 "빈 공간에 따라 모드 자동 선택"은 아직 미구현 — 지금은 **수동 토글**(SPEC 의 사용자 토글 부분).
- (추가 피드백) **포인트 1~2개면 자동 가운데 정렬** — `.payment-brands` 를 3열 grid → `flex; justify-content:center` + 카드 `flex-basis: calc((100%-20px)/3)`(3개 기준 폭 유지). 로고만 모드는 이미 center. **결제 안내 제목 편집** — 결제 단계 `#f_payTitle` 입력 → `state.payment.title` 즉시 반영(포인트 0개면 비활성). 검증 `scripts/verify-pay2.mjs` 6/6(좌우 여백 동일·제목 반영).

### 배경 이미지 토글 + 배경 시 유의사항 흰색 ("배경 있는 것/없는 것" 피드백)
하단 배경 사진이 `.x-banner.case-c` 에 하드코딩 → 케이스 고정이라 끄거나 다른 케이스에 켤 수 없었음. **토글로 분리** + 배경 위 유의사항 가독성 처리.
- `banner.css`: `.case-c` 에서 `background` 분리 → **`.x-banner.has-bg`** 로 이동(하단 50% 침실 그라데이션, [[feedback_xbanner_background_image]] 규칙 유지). 신규 `.has-bg .notices { color:#FFF; text-shadow }` — **배경 ON 시 유의사항 흰색**.
- `fit-banner.js` `applyBackground(banner, data)` — `data.background` 미설정 시 기본값(case-c만 ON), has-bg 클래스 토글. app.js rerender/openFullscreen/boot + preview-app.js(내보내기) 배선(6개 케이스 파일 미수정).
- 편집 UI: 기본정보 단계에 "배경 이미지" 체크박스(`#f_background`) — 켜면 흰 유의사항 안내. `state.background` 저장(자동저장·드래프트 포함).
- 검증(`scripts/verify-bg.mjs`): case-a 토글 ON→has-bg+유의사항 rgb(255,255,255)/OFF→해제·어두움, case-c 기본 ON+흰색 9/9. 내보내기 경로 풀 캡처로 침실 배경 위 흰 유의사항 가독 확인. verify-ux 24 회귀 없음.
- (확장 여지) 배경 이미지 2종(침실/쿠시노) 중 선택은 미구현 — 현재 침실 고정 on/off.
